GhostStack — Spec-Driven Autonomous Local Cloud Lab

Distributed-systems analysis of a 12-repository confederation that fuses local AWS emulation, multi-agent code synthesis, spec-driven planning, MCP-based tool federation, memory persistence, local inference, stealth browser automation, and document archival into a single coherent autonomous software factory. This dossier reasons across repositories as a unified system, not as independent packages.

1 · Executive Summary

Scope: cross-repository systems analysis · Audience: senior systems engineers, AI infrastructure researchers

GhostStack is not a product — it is an integration thesis. The thesis claims that twelve independently maintained open-source projects, when correctly composed, yield a complete cloud-native development substrate that operates with zero external SaaS dependency: spec-driven intent capture (spec-kit), multi-agent code generation (codebuff), free LLM routing (free-claude-code), local AWS emulation (floci), stealth E2E testing (CloakBrowser + Scrapling), persistent agent memory (claude-mem), local heavy-model review (AirLLM), tool federation (fastmcp), in-IDE retrieval (Vane), ML pipeline automation (ml-intern), and architectural archival (paperless-ngx).

Architectural Verdict
Coherent in principle, integration-heavy in practice. Three independent runtimes (JVM/GraalVM, Node/Bun, Python) must coexist. Coherence is achievable through MCP as a unifying control surface.
Strongest Property
Every plane (control, execution, persistence, inference) has a working open-source citizen. No theoretical components — every box is a real repo.
Weakest Property
No shared schema, no shared event bus, no shared identity model. Integration depends on per-tool MCP adapters and brittle glue scripts.

Top-Level Findings

  1. Two control planes compete: spec-kit (declarative intent → tasks) and codebuff's internal planner agent (imperative agent graph). A coordinator is required to prevent re-planning loops.
  2. Three runtime substrates must coexist: Quarkus/GraalVM native (floci), Node/TypeScript (codebuff, Vane), and Python (fastmcp, AirLLM, Scrapling, free-claude-code, ml-intern). IPC is HTTP/MCP — adequate but lossy for high-frequency events.
  3. Memory is currently single-agent-scoped: claude-mem persists per-session for Claude Code. There is no cross-agent shared semantic memory; this is the single largest architectural gap.
  4. Inference economics are uneven: free-claude-code provides 10 routed backends but is throughput-capped (NIM 40 req/min). AirLLM offers unlimited local 70B but only at ~1–3 tok/s on 4GB GPUs — viable for review, infeasible for inner-loop edits.
  5. Floci is the structural keystone: Its 47-service surface is the only component that anchors GhostStack to "real cloud" semantics. Every other repo could be swapped; floci cannot.

2 · GhostStack System Vision

GhostStack reframes the developer workstation as a self-contained autonomous cloud lab: the laptop is simultaneously the IDE, the orchestrator, the cloud, the test runner, the inference cluster, and the document archive. The system is designed around five axiomatic constraints:

Axiom 1 — Zero External Account
No AWS, no OpenAI, no Anthropic billing. Every dependency must be replaceable with a local or free-tier substitute. floci replaces AWS; free-claude-code replaces Anthropic; AirLLM/Ollama replace GPU vendors.
Axiom 2 — Spec-First, Code-Second
Code is downstream of specification. spec-kit's constitution → spec → plan → tasks pipeline is the canonical entrypoint. Agents do not improvise; they execute spec-derived tasks.
Axiom 3 — Multi-Agent over Monolithic LLM
No single LLM call is trusted with end-to-end coding. Tasks are decomposed across File Picker → Planner → Editor → Reviewer agents; each has narrow scope and verifiable output.
Axiom 4 — Production-Realistic Fidelity
Emulation must be high-fidelity enough that FLOCI_MODE=mirror can flip to real AWS with zero code change. Real Docker for Lambda; WAL-backed S3; DuckDB-backed Athena/Glue.
Axiom 5 — Memory as Infrastructure
Architectural decisions, specs, diagrams, and review notes are first-class artifacts. claude-mem persists agent state; paperless-ngx archives diagrams; Vane retrieves both.
Axiom 6 — MCP as Universal Bus
Heterogeneous runtimes (JVM/Node/Python) cannot share memory directly. MCP (Model Context Protocol) becomes the lingua franca: every repo is wrapped as MCP tools, resources, and prompts via fastmcp.

8 · Spec-Driven Development Analysis

Spec-kit imposes a four-phase pipeline: /speckit.constitution → /specify → /plan → /tasks → /implement. Each phase produces a versioned artifact under .specify/. GhostStack treats these artifacts as the canonical contract that downstream agents must satisfy.

Phase Artifacts
constitution.md — invariants, governance principles
spec.md — functional intent, user stories
plan.md — technical decomposition
tasks.md — ordered, dependency-aware unit work
Integration with codebuff
tasks.md is consumed as the codebuff Planner's input queue. Each task entry becomes a planner step; File Picker scopes each step to a file set. Codebuff's autonomous re-planning is constrained by spec-kit's task DAG — preventing runaway agent improvisation.

Spec → Code Coherence Inference

The spec-kit + codebuff coupling creates a two-tier planner: spec-kit operates at the macro level (feature decomposition), codebuff's Planner agent operates at the micro level (file-level edit sequencing). This is architecturally sound because the levels are non-overlapping. A risk emerges if codebuff's Planner revises the macro plan — this must be disallowed via prompt constraints or MCP-enforced scope.

Critical Coupling Point spec-kit emits tasks.md as markdown. codebuff consumes it via prompt injection. No formal schema validation exists. Drift in spec-kit's output format would silently break codebuff's Planner. A shared JSON-LD / OpenAPI contract should mediate this boundary.

10 · Multi-Agent Coordination Analysis

Codebuff's spawn_agents primitive supports hierarchical agent composition. Within GhostStack, five agent classes can be inferred:

File-scoped Editors
Read-write narrow context. Spawned per file by Planner. Conflict-prone on shared imports.
Reviewer Pool
Heterogeneous: cloud LLM for speed, AirLLM for sensitive paths, ml-intern for ML-specific review.
Verifier Agents
Drive CloakBrowser/Scrapling. Asynchronous to Editors. Emit pass/fail to Planner.
Knowledge Agents
Vane retrieval + claude-mem injection. Read-only against shared corpus.
Archivist Agents
Emit specs/diagrams to paperless-ngx. Idempotent. Tail of every pipeline.
Bootstrap Agent
spec-kit's task generator. Single-shot per phase.

Coordination Primitives Required

  • Mutex on file paths — needed to prevent two Editors writing the same file. Codebuff does not document filesystem-level locking; must be enforced by Planner sequencing or git working-tree partitioning.
  • Result aggregation — Reviewer outputs from heterogeneous reviewers must be merged. No documented aggregator exists; inferred to be Planner's responsibility.
  • Memory propagation — claude-mem hooks operate per-session; cross-session learning is via summarization. No shared agent memory exists.

11 · Browser Automation Layer Analysis

CloakBrowser and Scrapling form GhostStack's web-facing actuation surface. Their composition is non-trivial: CloakBrowser supplies the runtime (stealth Chromium with passing fingerprints); Scrapling supplies the adaptive parsing/selection layer and HTTP-based fetchers for non-JS endpoints.

CloakBrowser Role
Custom Chromium build with anti-detection patches. Exposes CDP over cloakserve. Wraps as Playwright/Stagehand-compatible. Used for E2E tests against deployed-to-floci apps and for high-fidelity simulation of real users against external services.
Scrapling Role
Adaptive selectors that survive DOM changes; StealthyFetcher (browser-backed) and HTTP fetchers (sub-ms); Cloudflare Turnstile bypass; spider framework for crawl-scale. In GhostStack, generates "sample apps" — i.e., ETL pipelines that scrape real data and target floci-emulated S3/DynamoDB.

Composition Pattern

       │
       ▼
  codebuff Editor writes ETL Lambda + S3 buckets
       │
       ▼
  Scrapling spider populates S3 via floci with REAL scraped data
       │
       ▼

       ▼
  E2E pass/fail ──► Reviewer ──► commit or re-plan
Security Surface Note CloakBrowser issues have flagged untrusted origins accessing local CDP can lead to unauthorized browser control and local file disclosure. In GhostStack this surface is exposed alongside floci :4566 and fcc-proxy :8082 — three privileged loopback services. Strict bind-to-127.0.0.1 and token auth are mandatory.

13 · Local Inference Layer Analysis

Two distinct inference modalities coexist: routed cloud-free (free-claude-code) and truly local heavy (AirLLM). Their roles are non-overlapping: free-claude-code serves the high-frequency inner loop (FilePicker/Planner/Editor turns); AirLLM serves the low-frequency, high-trust outer loop (Reviewer for sensitive paths).

free-claude-code (:8082)
FastAPI proxy emitting Anthropic-compatible responses. Backends: NVIDIA NIM (40 req/min), Kimi K2, DeepSeek, GLM, Qwen, Ollama, LM Studio, Cerebras Free, Together Free, Groq Free. Token-bucket + sliding-window rate-limit enforcement. Stateless. Hot-swappable backends.
AirLLM
Layer-wise inference: loads one transformer layer at a time onto GPU, swaps to CPU/disk for others. Enables 70B on 4GB VRAM without quantization. Throughput: ~1–3 tok/s on consumer hardware — usable for batch review, not for interactive coding.

16 · Runtime Lifecycle Analysis

Boot Order (topologically sorted)

  1. paperless-ngx + Postgres — slow startup; archive should be live before any artifacts are emitted.
  2. floci :4566 — Quarkus native, 24ms cold start. Must be live before any IaC is applied.
  3. free-claude-code :8082 — FastAPI, ~1s. Required by every agent.
  4. AirLLM — lazy-loaded on first Reviewer call; first call cold ~30–120s.
  5. Vane :3000 — Next.js + embedding model warm-up.
  6. fastmcp host — composite server; spawns child MCP servers; ~2–5s.
  7. CloakBrowser — lazy-launched per test; ~3s per profile.
  8. codebuff + spec-kit — CLI, on-demand.
  9. claude-mem — hook-driven; attaches to Claude Code session.

Shutdown Hazards

  • floci WAL must be flushed cleanly or S3-emulated state diverges from object layout on next boot.
  • Docker-backed Lambda containers leak if floci is killed via SIGKILL — Docker daemon retains zombies.
  • claude-mem hooks must be drained before session end or transcript chunks are lost.
  • paperless-ngx OCR workers must finish in-flight jobs; otherwise re-OCR on restart.

21 · Agent Runtime Analysis

Codebuff is a Node/TypeScript runtime. Its agents are TypeScript generators that interleave LLM calls with deterministic tool invocations. The runtime is single-process with cooperative concurrency. Agents do not have isolated address spaces — they share the codebuff session heap and the filesystem.

Agent Isolation
None at process level. Each agent is a generator function; isolation is by prompt scoping. Memory leaks across agents are possible via shared session state. Cooperative
Failure Containment
Agent throw → caught by runtime → reported to Planner. No retry semantics standardized. Manual
Tool Surface

23 · Bottleneck Analysis

  1. LLM throughput (inner loop) — NIM's 40 rpm dominates wall-clock. With 4 agents in parallel, effective ceiling is ~10 turns/min per agent. Mitigation: rotate to DeepSeek/Kimi during NIM cool-down.
  2. AirLLM token rate — 1–3 tok/s makes 70B local review feasible only for diffs <1k tokens. Larger diffs must fall back to cloud.
  3. Docker Lambda cold start — Real container cold starts (1–3s) dominate floci's emulated invocation latency. LRU warm-pool partially mitigates.
  4. OCR queue (paperless-ngx) — Single-worker default. Burst of diagram emissions can backlog.
  5. GPU contention — Single GPU shared by Vane embed, AirLLM, ml-intern. Sequential scheduling required.
  6. fastmcp tool latency aggregation — Each MCP hop adds 10–50ms. Pipelines of 5+ hops are perceptible.

24 · Scalability Analysis

GhostStack scales vertically (more CPU/RAM/GPU per host) cleanly. Horizontal scaling is not designed for — the architecture assumes a single developer workstation. Three dimensions exhibit hard limits:

  • Concurrent agents: codebuff is single-process; ~4–8 parallel agents before context-switch dominates.
  • Concurrent Lambdas: floci uses real Docker containers; bounded by host's Docker concurrency (~50–100 typical).
  • Concurrent E2E browsers: each CloakBrowser instance ~500MB; ~10–20 parallel on a 32GB host.
Production Trajectory Scaling GhostStack beyond a single host would require: (a) codebuff to be re-engineered for distributed agent execution, (b) floci to support a clustering mode, (c) a true shared event bus replacing in-process queues. Today's design is laptop-optimal, not cluster-optimal.

floci-io/floci — Local AWS Emulator (Quarkus Native)

Role: Execution Plane keystone · Language: Java (Quarkus) → GraalVM native
1. Repository Overview

Purpose: Drop-in replacement for AWS, exposing AWS SDK contracts on a single loopback endpoint (:4566). Emulates 47 services with real Docker for stateful runtimes (Lambda) and in-process implementations where fidelity allows.

Ecosystem role: The unique structural keystone — no other open-source project provides a free, feature-gate-less, GraalVM-native AWS emulator with this surface area. Direct competitor to LocalStack but explicitly without feature gates.

Maturity: Active (v1.5.9-class), single endpoint, documented 24ms cold start, 13 MiB idle.

Architectural category: Infrastructure abstraction / emulator / control-plane proxy.

2. Internal Architecture (Inferred)
  • HTTP Router: JAX-RS on top of Vert.x — non-blocking, reactive, single port :4566.
  • Service dispatch: Path/host header → service module. AWS region routing collapsed to local.
  • Storage backends (heterogeneous, per service):
    • S3 — write-ahead log (WAL) backed object store.
    • DynamoDB — hybrid in-memory + disk page cache.
    • SQS/SNS — in-memory queues with persistence checkpoints.
    • Lambda — Docker daemon spawns user-code containers; LRU warm pool retains N containers.
    • Athena/Glue — DuckDB sidecar process; SQL routed via JDBC-like bridge.
    • Bedrock Runtime — routed to local LLMs (Ollama) or external free providers.
  • Concurrency model: Vert.x event loop + worker pools. CPU-bound work (e.g., crypto for KMS) offloaded.
  • Native image: GraalVM ahead-of-time compilation; explains 24ms cold start.
3. Technology Stack
  • Quarkus framework, GraalVM native-image, JAX-RS, Vert.x, DuckDB (sidecar), Docker daemon (external dependency), Jackson, AWS SDK contract compliance.

Bedrock Runtime route is the AI-relevant surface: it proxies Bedrock-shaped requests to local LLMs (via fcc-proxy or Ollama). This lets ml-intern and deployed apps consume Bedrock Runtime APIs against a free backend.

A federation — not a platform. Each repository remains a sovereign project with its own release cadence, contributor community, and roadmap.   3. Reviewer reject ↔ Planner re-plan (unbounded loop)
  4. fcc-proxy ↔ external free providers (terms-of-service drift)
  5. CloakBrowser CDP ↔ untrusted local origins (security)

Likely Scaling Bottlenecks

  1. LLM throughput (NIM 40 rpm)
  2. AirLLM 1–3 tok/s on consumer GPU
  3. Docker Lambda cold-start fan-out
  4. Single-GPU contention across AirLLM/Vane/ml-intern
  5. fastmcp tool-call hop accumulation

Operational Feasibility

Feasible on a 32GB RAM / single mid-range GPU / 200GB free disk workstation. Feasible to bring up in a single Compose-file plus three native CLIs (codebuff, spec-kit, claude-mem on the host). Boot time ~30s warm. Daily operational burden: monitoring 14 long-running processes, rotating LLM backend keys, and managing GPU exclusivity.

Maintainability Outlook

Twelve independent maintainers means twelve risk vectors. Three (Scrapling, AirLLM, claude-mem) have single-maintainer bus-factor. floci is a heavyweight ongoing investment. The architecture itself is resilient to swap-outs at most planes — only floci is non-replaceable. With a pinned compatibility matrix and quarterly upgrade cadence, the federation is maintainable.

Production Realism Assessment

As a development lab — production-realistic. Developers can author, deploy, test, and iterate on full-fidelity cloud applications without an AWS account. As a production runtime — explicitly not. Mirror mode is the production bridge. The interesting question is whether floci's parity is good enough that mirror flips succeed without surprise. That requires a contract test suite that does not yet exist.

Overall Verdict

Verdict — GhostStack is Architecturally Viable as a Federation The composition is sound. Every plane has a real, working open-source owner. The federation pattern is realistic and the binding standard (MCP) is the correct bet. The remaining work is integration engineering — primarily fastmcp wrappers, a unified supervisor, a budget proxy, and a parity test suite for floci. These are tractable, not foundational, problems. GhostStack will not scale beyond a single workstation without re-architecture, but as a spec-driven autonomous local cloud lab for a single developer or small team, it is one of the most architecturally coherent open-source compositions currently assemblable.
Final Architectural Confidence B+ / 4.0 of 5.0. Strong on coherence, locality, and cost. Weak on cross-agent memory, mirror-mode parity verification, and multi-host scaling. None of the weaknesses are foundational; all are addressable with named, scoped work items.
